# Feature Builder - Поэтапное Создание Функций

## Назначение
Структурированный подход к созданию сложных функций и модулей. Разбивает большую задачу на управляемые этапы с проверками на каждом шаге.

## Промпт

```
Ты senior разработчик, создающий новую функциональность.

МЕТОДОЛОГИЯ ПОЭТАПНОЙ РАЗРАБОТКИ:

ШАГ 1: ПЛАНИРОВАНИЕ И АРХИТЕКТУРА
- Проанализируй требования
- Определи архитектурный подход
- Выбери паттерны проектирования
- Определи зависимости
- Покажи план разработки
- ЖДИ ПОДТВЕРЖДЕНИЯ перед продолжением

ШАГ 2: ТИПЫ И ИНТЕРФЕЙСЫ
- Определи все типы данных (TypeScript/Flow/PropTypes)
- Создай интерфейсы/контракты
- Опиши входные и выходные данные
- Документируй типы
- ЖДИ ПРОВЕРКИ типов

ШАГ 3: БАЗОВАЯ РЕАЛИЗАЦИЯ
- Реализуй happy path (основной сценарий)
- Пиши чистый, читаемый код
- Добавь подробные комментарии
- Следуй принципам SOLID
- ПОКАЖИ код для review

ШАГ 4: ОБРАБОТКА ОШИБОК
- Добавь обработку всех возможных ошибок
- Валидация входных данных
- Graceful degradation
- Логирование ошибок
- User-friendly сообщения об ошибках

ШАГ 5: ТЕСТИРОВАНИЕ
- Unit тесты для всех функций
- Integration тесты
- Edge cases
- Error cases
- Минимум 80% coverage

ШАГ 6: ДОКУМЕНТАЦИЯ
- JSDoc/docstrings для всех публичных методов
- README если это модуль
- Примеры использования
- Описание API

ШАГ 7: ОПТИМИЗАЦИЯ И REVIEW
- Code review чек-лист
- Проверка производительности
- Проверка безопасности
- Финальный рефакторинг

ПРАВИЛА:
- НЕ пропускай шаги
- НЕ переходи к следующему шагу без подтверждения
- Каждый шаг должен быть завершён полностью
- Если что-то непонятно - СПРОСИ
```

## Когда использовать

✅ **Используйте когда:**
- Создаёте новую сложную функцию
- Добавляете новый модуль
- Разрабатываете критичную логику
- Нужен контроль на каждом этапе
- Работаете с незнакомым доменом

❌ **Не используйте когда:**
- Простые CRUD операции
- Быстрое прототипирование
- Мелкие правки существующего кода
- Одноразовые скрипты

## Примеры использования

### Пример 1: Система аутентификации

**Запрос:**
```
Используя Feature Builder approach, создай модуль аутентификации
с JWT токенами, refresh tokens и role-based access control.
```

**Ответ AI - Шаг 1:**
```
ШАГ 1: ПЛАНИРОВАНИЕ И АРХИТЕКТУРА

ТРЕБОВАНИЯ:
- JWT для аутентификации
- Refresh tokens для продления сессии
- RBAC (Role-Based Access Control)
- Безопасное хранение токенов

АРХИТЕКТУРНЫЙ ПОДХОД:
1. Слой аутентификации (AuthService)
   - login/logout
   - token generation/validation

2. Middleware для защиты роутов
   - verifyToken
   - checkRole

3. Token хранилище (TokenStore)
   - save/retrieve/revoke tokens
   - Redis для production

4. RBAC модуль (RoleManager)
   - проверка прав
   - управление ролями

ЗАВИСИМОСТИ:
- jsonwebtoken (JWT операции)
- bcrypt (хеширование паролей)
- redis (хранение refresh tokens)

ПЛАН РАЗРАБОТКИ:
1. Типы и интерфейсы (User, Token, Role)
2. AuthService - core функционал
3. Middleware для Express/Fastify
4. RoleManager
5. Тесты
6. Документация

Одобряете архитектуру? Есть изменения?
```

### Пример 2: Система кэширования

**Запрос:**
```
Feature Builder: создай flexible caching module с поддержкой
множественных backend'ов (memory, Redis, filesystem)
```

**Ответ AI - Шаг 2 (Типы):**
```
ШАГ 2: ТИПЫ И ИНТЕРФЕЙСЫ

// Cache Provider Interface
interface CacheProvider {
  get<T>(key: string): Promise<T | null>;
  set<T>(key: string, value: T, ttl?: number): Promise<void>;
  delete(key: string): Promise<boolean>;
  clear(): Promise<void>;
  has(key: string): Promise<boolean>;
}

// Cache Configuration
interface CacheConfig {
  provider: 'memory' | 'redis' | 'filesystem';
  ttl?: number; // default TTL in seconds
  maxSize?: number; // for memory provider
  redisUrl?: string; // for redis provider
  cachePath?: string; // for filesystem provider
}

// Cache Entry (internal)
interface CacheEntry<T> {
  value: T;
  expiresAt: number;
  createdAt: number;
}

// Cache Statistics
interface CacheStats {
  hits: number;
  misses: number;
  size: number;
  hitRate: number;
}

ВХОДНЫЕ ДАННЫЕ:
- key: string - уникальный идентификатор
- value: any - данные для кэширования (JSON-serializable)
- ttl: number (optional) - время жизни в секундах

ВЫХОДНЫЕ ДАННЫЕ:
- get(): Promise<T | null> - данные или null
- set(): Promise<void> - void при успехе, Error при ошибке

Типы корректны? Что-то добавить?
```

## Модификации промпта

### Для TDD подхода
```
ИЗМЕНЕННЫЙ ПОРЯДОК:
1. Планирование
2. Типы и интерфейсы
3. СНАЧАЛА ТЕСТЫ (failing tests)
4. Реализация (пока тесты не пройдут)
5. Рефакторинг
6. Документация
```

### Для микросервисной архитектуры
```
ДОПОЛНИТЕЛЬНЫЕ ШАГИ:
- API contract (OpenAPI/gRPC)
- Service discovery integration
- Health checks
- Monitoring и metrics
- Graceful shutdown
```

### Для frontend компонентов
```
АДАПТАЦИЯ ДЛЯ UI:
1. Планирование: UX flow, state management
2. Типы: Props, State, Events
3. Mockup/Wireframe
4. Базовая разметка
5. Стили и адаптивность
6. Интерактивность и логика
7. Accessibility (a11y)
8. Storybook stories
9. Unit + E2E тесты
```

## Best Practices

1. **Не спешите:** Каждый шаг требует внимания
2. **Документируйте решения:** Почему выбрали этот подход
3. **Проверяйте на каждом шаге:** Легче исправить рано
4. **Думайте о будущем:** Расширяемость и поддержка
5. **Спрашивайте:** Если непонятно - уточняйте требования

## Чек-лист завершения

После выполнения всех шагов проверьте:

- [ ] Архитектура понятна и задокументирована
- [ ] Все типы определены и экспортированы
- [ ] Основная логика работает (happy path)
- [ ] Обработаны все error cases
- [ ] Тесты написаны и проходят (>80% coverage)
- [ ] API задокументирован (JSDoc/docstrings)
- [ ] Есть примеры использования
- [ ] Code review пройден
- [ ] Нет TODO/FIXME в production коде
- [ ] Performance приемлемый
- [ ] Security проверен

## Временные оценки

| Размер фичи | Время на Feature Builder |
|-------------|-------------------------|
| Малая (1 функция) | 30-60 минут |
| Средняя (модуль) | 2-4 часа |
| Большая (система) | 1-3 дня |

## Антипаттерны

❌ **Пропуск планирования:** "Давай сразу кодить"
✅ Потратьте 15% времени на планирование

❌ **Игнорирование типов:** "Добавим потом"
✅ Типы = контракт, определяйте сразу

❌ **Тесты в конце:** "Напишу когда всё работает"
✅ Тесты параллельно с разработкой

❌ **Пропуск документации:** "Код сам себя документирует"
✅ Хорошая документация = меньше вопросов
