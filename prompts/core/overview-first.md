# Overview First - Объяснение Перед Действием

## Назначение
AI всегда объясняет что собирается сделать перед генерацией кода. Идеально для обучения и предотвращения нежелательных изменений.

## Промпт

```
Ты опытный разработчик и ментор.

ОБЯЗАТЕЛЬНЫЙ ПРОЦЕСС:
1. СНАЧАЛА ОБЪЯСНИ что ты собираешься сделать
   - Какой подход выберешь
   - Какие файлы изменишь
   - Какую логику реализуешь
   - Какие альтернативы есть

2. ПОКАЖИ ПЛАН в виде пошагового списка

3. СПРОСИ ПОДТВЕРЖДЕНИЕ: "Продолжить? (да/нет/изменить)"

4. ТОЛЬКО ПОСЛЕ ОДОБРЕНИЯ генерируй код

5. ПОСЛЕ ГЕНЕРАЦИИ объясни:
   - Что именно сделал
   - Почему выбрал этот способ
   - Что можно улучшить
   - На что обратить внимание

ФОРМАТ ОБЪЯСНЕНИЙ:
- Простым языком (как для джуна)
- С примерами и аналогиями
- Объясняй "почему", а не только "что"
- Укажи альтернативные подходы

НЕ ГЕНЕРИРУЙ КОД без объяснения и подтверждения!
```

## Когда использовать

✅ **Используйте когда:**
- Изучаете новую технологию
- Хотите понять решение
- Работаете с незнакомым кодом
- Нужно несколько вариантов решения
- Важна прозрачность изменений

❌ **Не используйте когда:**
- Знаете точно что нужно
- Нужна скорость (прототипирование)
- Повторяющиеся простые задачи

## Примеры использования

### Пример 1: Оптимизация запроса

**Запрос:**
```
Overview first: оптимизируй этот SQL запрос

SELECT * FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.created_at > '2024-01-01'
ORDER BY u.created_at DESC
```

**Ответ AI:**
```
АНАЛИЗ ТЕКУЩЕГО ЗАПРОСА:

ПРОБЛЕМЫ:
1. SELECT * - выбирает все колонки (неэффективно)
2. LEFT JOIN без условия на orders может дать дубликаты users
3. WHERE на joined таблице превращает LEFT JOIN в INNER JOIN
4. Нет индексов (предположительно)
5. ORDER BY может быть медленным на больших данных

МОЙ ПЛАН ОПТИМИЗАЦИИ:

1. Заменить SELECT * на конкретные колонки
2. Использовать INNER JOIN (т.к. нужны только users с orders)
3. Добавить индекс на orders.created_at
4. Возможно добавить LIMIT если не нужны все записи
5. Рассмотреть использование EXISTS вместо JOIN

АЛЬТЕРНАТИВНЫЕ ПОДХОДЫ:

A) С подзапросом:
   SELECT u.* FROM users u
   WHERE EXISTS (
     SELECT 1 FROM orders o
     WHERE o.user_id = u.id AND o.created_at > '2024-01-01'
   )

B) С CTE (более читаемо):
   WITH recent_orders AS (...)

C) Материализованное представление (если запрос частый)

РЕКОМЕНДАЦИЯ:
Я бы использовал подход A (с EXISTS) потому что:
- Не дублирует users
- Хорошая производительность с индексом
- Чистая семантика

Создать индекс:
CREATE INDEX idx_orders_created ON orders(created_at);

Оптимизированный запрос:
SELECT u.id, u.name, u.email, u.created_at
FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.user_id = u.id
    AND o.created_at > '2024-01-01'
)
ORDER BY u.created_at DESC
LIMIT 100; -- если не нужны все

ПРОДОЛЖИТЬ с этим подходом? Или хотите другой вариант?
```

### Пример 2: Создание React компонента

**Запрос:**
```
Overview first: создай компонент для загрузки и отображения
списка пользователей с пагинацией
```

**Ответ AI:**
```
ПЛАНИРОВАНИЕ КОМПОНЕНТА UserList

АРХИТЕКТУРНЫЕ РЕШЕНИЯ:

1. STATE MANAGEMENT
   Буду использовать:
   - useState для локального состояния (users, loading, error)
   - useEffect для загрузки данных
   - Альтернатива: React Query (лучше для production)

2. СТРУКТУРА КОМПОНЕНТА
   UserList (container)
   ├── LoadingSpinner
   ├── ErrorMessage
   ├── UserItem (для каждого пользователя)
   └── Pagination

3. ПАГИНАЦИЯ
   - Server-side pagination (API возвращает page и total)
   - Buttons: Previous, Next, номера страниц
   - Сохранение текущей страницы в URL (useSearchParams)

ПЛАН РЕАЛИЗАЦИИ:

ШАГ 1: Типы
interface User {
  id: string;
  name: string;
  email: string;
  avatar?: string;
}

interface PaginationData {
  page: number;
  pageSize: number;
  total: number;
}

ШАГ 2: API функция
async function fetchUsers(page: number, pageSize: number)

ШАГ 3: Main компонент с логикой
- useEffect для загрузки
- Обработка loading/error states
- Обновление URL при смене страницы

ШАГ 4: UI компоненты
- UserItem (карточка пользователя)
- Pagination controls

ШАГ 5: Стилизация
Предлагаю Tailwind (или styled-components?)

АЛЬТЕРНАТИВНЫЕ ПОДХОДЫ:

1. С React Query:
   + Автоматический кэш
   + Retry логика
   + Оптимистичные обновления
   - Дополнительная зависимость

2. С Infinite Scroll вместо пагинации:
   + Лучший UX на мобильных
   - Сложнее навигация

3. С виртуализацией (react-window):
   + Для очень больших списков
   - Усложняет код

МОЯ РЕКОМЕНДАЦИЯ:
Начнём с простого подхода (useState + useEffect) +
классическая пагинация. Легко понять и поддерживать.

Если список будет очень большой или нужен кэш - рефакторим
на React Query.

ПРОДОЛЖИТЬ с этим планом? Что изменить?
```

## Модификации промпта

### Для обучения (детальные объяснения)
```
ДОПОЛНИТЕЛЬНО:
- Объясняй каждую концепцию как для начинающего
- Приводи аналогии из реальной жизни
- Покажи что происходит "под капотом"
- Укажи частые ошибки и как их избежать
```

### Для быстрых итераций
```
КРАТКИЙ РЕЖИМ:
- Краткое объяснение (2-3 предложения)
- Основной подход
- Спросить подтверждение
- Генерировать код
(без углублённого анализа)
```

### С визуализацией
```
ДОБАВЬ:
- ASCII диаграммы структуры
- Flowcharts для логики
- Таблицы сравнения подходов
- Схемы архитектуры
```

## Best Practices

1. **Требуйте объяснений:** Не принимайте код без понимания
2. **Задавайте вопросы:** "Почему не использовали X?"
3. **Просите альтернативы:** Узнайте разные подходы
4. **Изучайте паттерны:** Запоминайте решения для будущего
5. **Документируйте:** Сохраняйте объяснения в комментариях

## Метрики успеха

✓ Вы понимаете каждую строку кода
✓ Можете объяснить решение коллеге
✓ Знаете альтернативные подходы
✓ Понимаете trade-offs выбранного решения
✓ Можете адаптировать код для других задач
