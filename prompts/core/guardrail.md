# Guardrail Prompt - Контроль Изменений

## Назначение
Обеспечивает точечные изменения кода без незапрошенных модификаций. Критически важен для работы с существующим кодом.

## Промпт

```
Ты опытный разработчик, работающий с существующим кодом.

СТРОГИЕ ПРАВИЛА:
1. Вноси ТОЛЬКО те изменения, которые явно запрошены
2. НЕ изменяй форматирование, если это не запрошено
3. НЕ рефактори код, если это не часть задачи
4. НЕ меняй названия переменных/функций без запроса
5. НЕ добавляй новые зависимости без согласования

ПРОЦЕСС:
1. Проанализируй запрос и определи минимальный набор изменений
2. Покажи diff/план изменений до применения
3. Спроси подтверждение
4. Примени ТОЛЬКО согласованные изменения
5. Объясни, что именно изменилось и почему

ВАЖНО: Если тебе кажется, что нужны дополнительные изменения -
СПРОСИ разрешение, не делай их автоматически.
```

## Когда использовать

✅ **Используйте когда:**
- Работаете с production кодом
- Нужно внести точечные изменения
- Важна стабильность существующего кода
- Работаете в команде (избегайте конфликтов)
- Делаете bug fix без рефакторинга

❌ **Не используйте когда:**
- Пишете новый код с нуля
- Делаете большой рефакторинг
- Прототипируете

## Примеры использования

### Пример 1: Исправление бага

**Запрос:**
```
Используя guardrail prompt, исправь бесконечный цикл в функции fetchData.
Файл: src/api/client.ts, строка 45-60
```

**Ответ AI:**
```
Я нашёл бесконечный цикл в строке 52. Проблема в условии while(true)
без break условия.

ПЛАН ИЗМЕНЕНИЙ:
- Строка 52: добавить счётчик попыток
- Строка 56: добавить проверку максимального количества попыток
- Строка 58: добавить break при достижении лимита

Это минимальные изменения для исправления бага.
Применить? (да/нет)
```

### Пример 2: Добавление параметра

**Запрос:**
```
С guardrail: добавь опциональный параметр timeout в функцию apiCall,
значение по умолчанию 5000мс
```

**Ответ AI:**
```
ИЗМЕНЕНИЯ:
1. Добавлю timeout: number = 5000 в параметры функции
2. Передам timeout в axios.request в поле config

Других изменений не будет. Форматирование и логика остаются прежними.
Продолжить?
```

## Модификации промпта

### Для критичного кода
Добавьте:
```
ДОПОЛНИТЕЛЬНО:
- Покажи unit тесты, которые могут сломаться
- Проверь обратную совместимость
- Укажи потенциальные риски
```

### Для работы с legacy кодом
Добавьте:
```
LEGACY CODE MODE:
- Сохраняй существующий стиль кода (даже если он не современный)
- Не обновляй deprecated API без явного запроса
- Минимизируй изменения в зависимостях
```

## Troubleshooting

**Проблема:** AI всё равно меняет слишком много

**Решение:**
```
СТОП. Ты изменил больше, чем я просил.
Откати все изменения и сделай ТОЛЬКО:
[конкретный список изменений]
```

**Проблема:** AI спрашивает разрешение на каждую мелочь

**Решение:**
```
Ты можешь вносить изменения в пределах [конкретной функции/файла]
без дополнительных запросов. Но предупреди, если нужно изменить API.
```

## Best Practices

1. **Будьте специфичны:** Укажите файл, функцию, строки
2. **Просите diff:** Всегда просите показать изменения до применения
3. **Контролируйте scope:** Явно ограничьте область изменений
4. **Проверяйте:** Всегда review сгенерированный код
5. **Итерируйте:** Если результат не тот - уточните guardrails

## Метрики успеха

✓ Изменено только то, что просили
✓ Существующая функциональность не сломалась
✓ Тесты проходят
✓ Нет незапланированных side effects
✓ Code review проходит быстро
